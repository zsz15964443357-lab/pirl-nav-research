# 09 风险清单

## 风险 1：项目变成系统拼装

表现：方案被写成语义分割 + PyBullet + PPO + ROS2 的组合。

规避：所有文档都要回到 action-conditioned predictive intent-risk 和 constrained RL。

## 风险 2：场景偏向方法

表现：场景只对 PIRL-Nav 有利，baseline 没有公平机会。

规避：加入 dynamic_crossing、semantic_distractor 和固定测试 seed；每个场景可视化审查。

## 风险 3：策略过度保守

表现：near miss 降低，但 detour、time-to-goal、jerk 明显恶化。

规避：主表必须报告 detour 和 jerk；semantic_distractor 必须保留。

## 风险 4：结果依赖 safety shield

表现：policy+shield 很好，但 policy-only 很差。

规避：报告 SDI、intervention count、intervention norm、policy-only 与 policy+shield。

## 风险 5：训练不稳定

表现：PPO-Lagrangian λ 振荡、策略停滞或过度保守。

规避：先训练 PPO 和 PPO+TTC；约束先只保留 C_near 与 C_risk；稳定后再加入 C_shield。

## 风险 6：感知过于理想

表现：训练和测试都使用 oracle state，缺少噪声。

规避：加入 semantic noise、tracking noise、dropout、detection delay、control disturbance。

## 风险 7：ROS2 验证夸大

表现：只报告平均延迟，声称硬实时。

规避：只说实时友好验证；报告 mean / p95 / p99 / max latency 与 deadline miss。

## 风险 8：Codex CLI 过度实现

表现：跳过审查直接生成复杂代码或训练脚本。

规避：一次只给一个 task 文件；每阶段必须通过网页端审查。
