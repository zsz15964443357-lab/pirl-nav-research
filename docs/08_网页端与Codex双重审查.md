# 08 网页端与 Codex 双重审查

## 目标

本项目采用网页端和 Codex CLI 双重审查，避免 Codex CLI 在没有研究判断的情况下过度实现、偏离主线或浪费额度。

## 审查角色

### 网页端审查

负责研究判断：

- 场景是否合理；
- 风险逻辑是否成立；
- 方法是否仍围绕论文主线；
- 指标是否能支撑贡献点；
- rollout 行为是否可信。

### Codex CLI 审查

负责工程判断：

- 代码是否结构清楚；
- 测试是否通过；
- 输出是否可复现；
- 是否遵守当前 task 边界；
- 是否产生可审查材料。

## 每阶段审查流程

```text
1. 网页端确认阶段目标
2. Codex CLI 读取单个 task 文档执行
3. Codex CLI 输出代码、测试、可视化或日志
4. 网页端审查研究合理性
5. 如需修正，Codex CLI 只修当前阶段问题
6. 通过后进入下一阶段
```

## PASS / FIX / REJECT

每个阶段的结论只能是：

- PASS：进入下一阶段；
- FIX：保留当前阶段，修正后重审；
- REJECT：放弃当前实现路线，回到方案设计。

## 关键审查点

### 场景阶段

重点看布局、轨迹、遮挡、难度、可达性、是否存在必撞或无风险问题。

### 训练阶段

重点看行为是否合理，不能只看数值。必须审查 rollout GIF 或轨迹图。

### PIRL-Nav 阶段

重点看提前避险是否真的来自 intent-risk，而不是来自过度保守或 safety shield。

### ROS2 阶段

重点看延迟、消息对齐、指令堆积、policy_cmd 与 safe_cmd 的差异。
